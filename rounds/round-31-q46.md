## Round 31 — Q46: GitHub CI/CD and Supply Chain Security (Mix & SLSA)

### DeepSeek

**Q46.1 — GitHub Actions for Elixir/Mix: Should we adopt it?** GitHub’s "Build and test" workflow is the standard for most Elixir projects. However, for `vaglio`, we must evaluate it against our "sovereign hosting" and "forkable context" goals.
- **Advantage:** Low friction. It provides immediate feedback on PRs.
- **Risk:** It reinforces the dependency on GitHub’s infrastructure. If the orchestrator’s CI only runs on GitHub Actions, then a fork on a different platform (like a local Forgejo instance) loses its validation gate.
- **Recommendation:** We should use the Mix workflow but **wrap it around our Nix flake**. Since we already have a `flake.nix` in the repo, the GitHub Action should simply run `nix flake check`. This ensures that the CI logic is repo-resident and portable, even if the runner is GitHub. `[inferred]`

[satisfied]

**Q46.2 — SLSA Generic Generator:** SLSA (Supply-chain Levels for Software Artifacts) is about provenance. It provides a cryptographically signed record of *how* an artifact was built.
- **For `vaglio`:** As an orchestrator that makes design decisions, our "artifact" isn't just a binary—it's the `DECISION.md` and the `RoundRun` state.
- **Verdict:** Implementing the SLSA generator for our Elixir releases is good hygiene, but it doesn't solve our core problem: "provenance of thought." We should prioritize signing our `dolt` commits over signing our OTP releases. `[observed]`

[satisfied]

### Codex

**Q46.1 — Portable CI Architecture:** We should implement a `justfile` or a `Makefile` that wraps `nix flake check`. The GitHub Action should then just call `just ci`. This ensures that the same CI that runs on GitHub can be run locally by an agent in an LXC or by a developer in a terminal.
- **Constraint:** We must ensure the `GH_TOKEN` and other secrets don't leak into the SLSA provenance record. `[observed]`

[satisfied]

**Q46.2 — SLSA and Dolt:** Since we are pivoting to Dolt (Round 30), our SLSA strategy should include the database. If we release "pre-baked" discussion databases, they *must* have SLSA provenance. Users shouldn't just "trust" a binary blob of a discussion; they should have a signed record that it was generated by the `vaglio` orchestrator from a specific `BRIEF.md`. `[inferred]`

[satisfied]
